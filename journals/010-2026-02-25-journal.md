## Key Insights

### Self-pipe design: why it works without NSRunLoop
`addObserverForName:object:queue:usingBlock:` with `queue:nil` delivers notifications synchronously on the posting thread (whatever thread EventKit fires the notification on). The block only calls `write(2)` — a pure C syscall, safe from any thread. The Go goroutine blocks on `f.Read()` at the other end of the pipe. No run loop needed, no `//export`, no `runtime.LockOSThread`.

### goroutine double-channel design in bridge_darwin.go
The `WatchChanges` implementation uses a two-layer channel approach:
```go
inner := watchChangesFromFile(ctx, f)
for range inner {
    select {
    case ch <- struct{}{}:
    default:
    }
}
```
`watchChangesFromFile` does the raw pipe reading and ctx cancellation handling. The outer goroutine handles the coalescing `select`-with-default and owns the C cleanup via `defer`. This separation means the `watchChangesFromFile` helper is pure Go (testable without cgo) while the cgo lifecycle is confined to `bridge_darwin.go`.

### Do NOT call f.Close()
`os.NewFile(uintptr(fd), name)` does NOT take fd ownership. The fd lifecycle belongs to `ek_cal_watch_stop` / `ek_rem_watch_stop` which calls `close(2)` on both pipe ends. If Go's `f.Close()` were called, it would double-close the fd, causing undefined behavior on any subsequent `open(2)` that reuses the fd number.

### FD_CLOEXEC on both pipe ends
Both read and write ends get `fcntl(fd, F_SETFD, FD_CLOEXEC)` to prevent fd inheritance by any child process (e.g., `osascript` calls, shell subprocesses).

### ctx cancellation race in tests
The ctx cancel test has a subtle race: the goroutine may have already read a byte from the pipe and sent a signal BEFORE `cancel()` takes effect. The test drains one potential pending signal before checking for channel close. This avoids a flaky test without changing the API semantics.

### Package-level mutex, not Client-level
`calWatchMu` / `remWatchMu` are package-level, not on the `Client` struct. Rationale: the `EKEventStore` singleton is package-level (one per process per package). Having two `Client` instances from `calendar.New()` and one calling `WatchChanges` while the other tries to also call it should be an error — they'd both be observing the same store. Package-level mutex enforces this correctly.

### IDE false positives on cgo files
The IDE flagged `"context" imported and not used` in `bridge_darwin.go` when running in Linux mode. This is a false positive — the IDE was checking the `!darwin` build constraint path. The `context` import IS used by `WatchChanges` on Darwin. Verified by `go build ./...` (darwin, exit 0) and `GOOS=linux CGO_ENABLED=0 go build ./...` (exit 0).

### Coverage ceiling maintained
New `WatchChanges` in `bridge_darwin.go` is unreachable by `go test` (same cgo boundary). The `watchChangesFromFile` helper in `watch.go` is fully testable. Coverage dropped slightly: calendar 56.7% → 54.4%, reminders 54.9% → 52.8%. This is expected: new cgo functions in bridge_darwin.go added uncoverable statements. All testable code (watch.go, parse.go) at ~100%.

## Decisions Made

1. **`watchChangesFromFile` in separate non-cgo file**: Extracted to `watch.go` (no build tag) so unit tests can inject `os.Pipe()` without any cgo dependency. Mirror of the `parse.go` pattern for JSON parsing.

2. **Two-layer channel architecture**: Outer goroutine in `bridge_darwin.go` owns cgo lifecycle (start/stop C functions, mutex). Inner `watchChangesFromFile` goroutine owns pipe reading and ctx handling. Clean separation of concerns.

3. **Integration tests added to existing scripts**: Tests 32–35 (calendar) and 31–34 (reminders) added inline in the existing integration scripts rather than a new file. Consistent with existing test numbering and style.

4. **Static globals in ObjC** (`ek_watch_pipe`, `ek_store_observer`): Package-level Go mutex prevents double-start from Go side, so idempotent check `if (ek_watch_pipe[0] != -1) return 1` in `ek_cal_watch_start` is a belt-and-suspenders guard. Globals are static (file-scoped in C), consistent with existing singleton pattern.

5. **No `ErrWatcherActive` sentinel**: Used `errors.New("calendar: watcher already active")` inline per PRD guidance. Add sentinel var only if fan-out use case emerges.

## Key Insights

### Why cross-process notifications didn't work

`EKEventStoreChangedNotification` has two delivery paths:
1. **Intra-process**: Posted via `NSNotificationCenter` synchronously on the EventKit thread. `queue:nil` means the block fires on that same thread immediately — no run loop needed.
2. **Cross-process**: Delivered via Darwin's distributed notification / mach port infrastructure, which requires an active **main thread CFRunLoop**. Background threads — including all goroutine OS threads — do NOT work.

### Failed fix attempts (first session, before rollback)

**Attempt 1: Dedicated NSThread running NSRunLoop** — `ek_cal_watch_start` spawned an `EKCalWatchThread` that ran `[[NSRunLoop currentRunLoop] run]` and registered the observer inside `threadMain`. Proved via ObjC probe: background `NSThread` run loops don't receive cross-process `EKEventStoreChangedNotification`.

**Attempt 2: `ek_cal_run_loop_tick()` from goroutine** — Added `CFRunLoopRunInMode(kCFRunLoopDefaultMode, 0, true)` called from a Go goroutine every 50ms. Didn't work: cgo calls run on goroutine OS threads, not the OS main thread. `CFRunLoopRunInMode` pumps the run loop for *that thread*, not the main thread.

### Correct pattern: consumer owns main thread run loop

The library itself is correct for intra-process use. Cross-process is a macOS constraint the consumer must handle:

```go
// In main():
runtime.LockOSThread()  // pins main goroutine to OS main thread

// All Go logic in separate goroutine
go func() {
    for range changes { /* diff and print */ }
}()

// Main goroutine pumps CFRunLoop (via its own cgo block)
for {
    C.pump_run_loop(0.1)  // CFRunLoopRunInMode(kCFRunLoopDefaultMode, 0.1, false)
    select {
    case <-done: return
    default:
    }
}
```

The consumer script has its own `#cgo LDFLAGS: -framework CoreFoundation` — the library does NOT need CoreFoundation linked.

## Decisions Made

1. **Library does NOT pump the run loop**: The `WatchChanges` API documents that cross-process consumers must handle CFRunLoop themselves. The library works correctly for the primary use case (same-process writes + watch). Adding run loop management to the library would force all consumers into a main-thread constraint even for intra-process use.

2. **consumer/main.go has its own cgo block**: `pump_run_loop` is a 3-line C function in the consumer binary. No new exports from the library. `-framework CoreFoundation` only in the consumer's LDFLAGS.

3. **No tick functions in library**: Reverted all `ek_cal_run_loop_tick` / `ek_rem_run_loop_tick` additions — they were the wrong abstraction. The library stays clean.

4. **PRD status stays "Planned" in the file**: The PRD documents the design; CLAUDE.md tracks completion status. No need to edit the PRD itself to say "done".
