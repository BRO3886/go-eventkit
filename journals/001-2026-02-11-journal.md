## Key Insights

### NSColor requires AppKit, not just Foundation
Calendar colors come back as `NSColor` objects. Initially tried using CoreGraphics directly but `cal.color` returns `NSColor`. Had to add `#import <AppKit/AppKit.h>` and `-framework AppKit` to LDFLAGS. The color-to-hex conversion requires `NSColorSpace.sRGBColorSpace` conversion first.

### TCC calendar access requires app restart
When granting calendar access to a terminal app (Cursor), macOS requires the app to quit and reopen. `tccutil reset Calendar` doesn't bypass this. The TCC dialog appears but access isn't effective until restart.

### Coverage ceiling with cgo
Unit test coverage tops out at 58.9% because cgo bridge functions (`bridge_darwin.go`) can't be tested without real EventKit. All testable code (types, parsing, marshaling in `calendar.go` and `parse.go`) achieves 100% coverage. The mock tests in `bridge_mock_test.go` test the JSON contract thoroughly without needing cgo.

### ISO 8601 parsing needs multiple format fallbacks
EventKit returns dates in varying formats. The parser handles three: with fractional seconds (`2006-01-02T15:04:05.000Z`), without (`2006-01-02T15:04:05Z`), and RFC3339 with offset (`2006-01-02T15:04:05-07:00`).

### File naming gotcha with Go test runner
Named integration test file `integration_test.go` initially â€” Go treats `_test.go` files specially and won't `go run` them. Renamed to `integration.go` with a build tag instead.

### `interface{}` -> `any` linting
Go 1.18+ prefers `any` over `interface{}`. IDE diagnostics flagged this across test files. Applied `replace_all` fix.

## Decisions Made
- **JSON bridge format**: ObjC returns JSON strings via `char*`, Go parses into typed structs. Keeps C interface minimal (just strings + ints), avoids complex struct marshaling across cgo boundary.
- **Separate parse.go**: Extracted all JSON parsing/marshaling into `parse.go` (no build tags) so it's fully testable without cgo. `bridge_darwin.go` only contains cgo wrapper functions.
- **Functional options for queries**: `WithCalendar(name)`, `WithCalendarID(id)`, `WithSearch(query)` â€” idiomatic Go, extensible without breaking changes.
- **Pointer fields in UpdateEventInput**: `*string`, `*time.Time` etc. allow distinguishing "not set" from "clear this field" (empty string clears, nil skips).
- **eventIdentifier over calendarItemIdentifier**: Stable across recurrence edits per Apple docs.
- **internal/store/ kept but unused**: Calendar package has its own inline EKEventStore singleton in bridge_darwin.m. The internal/store/ package exists for future sharing with reminders/ package (Phase 2).

## Session 2 - Integration Testing & Bug Fixes

## Goal
Run integration tests against real macOS EventKit (after Cursor restart for TCC calendar access), fix any bugs found, and verify full end-to-end CRUD.

## What Changed
- Fixed `CalendarTypeSubscription` enum: was 5, corrected to 3 (Apple's `EKCalendarType` enum order: Local=0, CalDAV=1, Exchange=2, **Subscription=3**, Birthday=4)
- Fixed `readOnly` JSON serialization in `bridge_darwin.m`: `@(!cal.allowsContentModifications)` produced integer 0/1 instead of JSON boolean. Changed to `cal.allowsContentModifications ? @NO : @YES` which produces proper `true`/`false`
- Updated test assertions to match corrected enum values
- Ran full integration test suite: 17/17 passing

## Key Insights

### ObjC `@(BOOL)` doesn't always produce JSON booleans
`@(!cal.allowsContentModifications)` boxes the result of `!BOOL` as `NSNumber` with integer value 0 or 1. `NSJSONSerialization` serializes this as `0`/`1`, not `true`/`false`. Fix: use ternary with explicit `@YES`/`@NO` literals, which `NSJSONSerialization` correctly serializes as JSON booleans.

```objc
// BAD: produces 0/1 in JSON
d[@"readOnly"] = @(!cal.allowsContentModifications);

// GOOD: produces true/false in JSON
d[@"readOnly"] = cal.allowsContentModifications ? @NO : @YES;
```

### EKCalendarType enum ordering
Apple's `EKCalendarType` is: Local=0, CalDAV=1, Exchange=2, **Subscription=3**, Birthday=4. The PRD had Birthday=4 and Subscription=5 which was wrong â€” Subscription comes before Birthday. This caused subscribed calendars (Formula 1, US Holidays) to show as "unknown" type.

### Integration test results confirm EventKit sees all accounts
EventKit successfully reads calendars from iCloud (Home, Work, Family, Flights), Google/CalDAV (Gatech, siddverma1999@gmail.com, Holidays in India), subscriptions (Formula 1, US Holidays), and system (Birthdays). This is more complete than AppleScript.

## Decisions Made
- **Coverage stays at 58.9%**: The uncovered 41% is exclusively `bridge_darwin.go` cgo wrapper functions. These call real C functions and can't be unit tested without mocking cgo (impractical). Integration tests cover this path instead.
