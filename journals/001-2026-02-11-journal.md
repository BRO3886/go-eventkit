## Key Insights

### NSColor requires AppKit, not just Foundation
Calendar colors come back as `NSColor` objects. Initially tried using CoreGraphics directly but `cal.color` returns `NSColor`. Had to add `#import <AppKit/AppKit.h>` and `-framework AppKit` to LDFLAGS. The color-to-hex conversion requires `NSColorSpace.sRGBColorSpace` conversion first.

### TCC calendar access requires app restart
When granting calendar access to a terminal app (Cursor), macOS requires the app to quit and reopen. `tccutil reset Calendar` doesn't bypass this. The TCC dialog appears but access isn't effective until restart.

### Coverage ceiling with cgo
Unit test coverage tops out at 58.9% because cgo bridge functions (`bridge_darwin.go`) can't be tested without real EventKit. All testable code (types, parsing, marshaling in `calendar.go` and `parse.go`) achieves 100% coverage. The mock tests in `bridge_mock_test.go` test the JSON contract thoroughly without needing cgo.

### ISO 8601 parsing needs multiple format fallbacks
EventKit returns dates in varying formats. The parser handles three: with fractional seconds (`2006-01-02T15:04:05.000Z`), without (`2006-01-02T15:04:05Z`), and RFC3339 with offset (`2006-01-02T15:04:05-07:00`).

### File naming gotcha with Go test runner
Named integration test file `integration_test.go` initially — Go treats `_test.go` files specially and won't `go run` them. Renamed to `integration.go` with a build tag instead.

### `interface{}` -> `any` linting
Go 1.18+ prefers `any` over `interface{}`. IDE diagnostics flagged this across test files. Applied `replace_all` fix.

## Decisions Made
- **JSON bridge format**: ObjC returns JSON strings via `char*`, Go parses into typed structs. Keeps C interface minimal (just strings + ints), avoids complex struct marshaling across cgo boundary.
- **Separate parse.go**: Extracted all JSON parsing/marshaling into `parse.go` (no build tags) so it's fully testable without cgo. `bridge_darwin.go` only contains cgo wrapper functions.
- **Functional options for queries**: `WithCalendar(name)`, `WithCalendarID(id)`, `WithSearch(query)` — idiomatic Go, extensible without breaking changes.
- **Pointer fields in UpdateEventInput**: `*string`, `*time.Time` etc. allow distinguishing "not set" from "clear this field" (empty string clears, nil skips).
- **eventIdentifier over calendarItemIdentifier**: Stable across recurrence edits per Apple docs.
- **internal/store/ kept but unused**: Calendar package has its own inline EKEventStore singleton in bridge_darwin.m. The internal/store/ package exists for future sharing with reminders/ package (Phase 2).

## Key Insights

### ObjC `@(BOOL)` doesn't always produce JSON booleans
`@(!cal.allowsContentModifications)` boxes the result of `!BOOL` as `NSNumber` with integer value 0 or 1. `NSJSONSerialization` serializes this as `0`/`1`, not `true`/`false`. Fix: use ternary with explicit `@YES`/`@NO` literals, which `NSJSONSerialization` correctly serializes as JSON booleans.

```objc
// BAD: produces 0/1 in JSON
d[@"readOnly"] = @(!cal.allowsContentModifications);

// GOOD: produces true/false in JSON
d[@"readOnly"] = cal.allowsContentModifications ? @NO : @YES;
```

### EKCalendarType enum ordering
Apple's `EKCalendarType` is: Local=0, CalDAV=1, Exchange=2, **Subscription=3**, Birthday=4. The PRD had Birthday=4 and Subscription=5 which was wrong — Subscription comes before Birthday. This caused subscribed calendars (Formula 1, US Holidays) to show as "unknown" type.

### Integration test results confirm EventKit sees all accounts
EventKit successfully reads calendars from iCloud (Home, Work, Family, Flights), Google/CalDAV (Gatech, siddverma1999@gmail.com, Holidays in India), subscriptions (Formula 1, US Holidays), and system (Birthdays). This is more complete than AppleScript.

## Key Insights

### `isFlagged` does not exist on EKReminder
Exact same lesson from rem Session 3 journal. Used `r.isFlagged` in bridge_darwin.m which doesn't compile. EventKit does not expose the "flagged" property. Hardcoded to `@NO` with a comment. The `Flagged` field in Go type is kept but will always be false from EventKit.

### Default reminder list name varies per user
Integration test hardcoded `"Reminders"` as the list name — failed because this user's default list is `"Inbox"`. Fixed by dynamically using the first list returned by `Lists()`.

### rem's `pkg/client` DOES build on linux
User said "it should not build for linux as ../rem does not" — but verified that `GOOS=linux CGO_ENABLED=0 go build ./pkg/client/...` works fine in rem. The CLI binary fails (imports darwin-only `applescript` package), but the library packages compile via stubs. go-eventkit's current pattern (bridge_other.go stubs) is correct and matches rem.

### `__thread` TLS error pattern is a race condition
Research agent found that the `static __thread char* last_error` pattern used in both calendar and reminders bridges has a subtle race: cgo can reschedule goroutines to different OS threads between the error-setting call and the error-reading call. Fix would be to return errors inline via a C struct. Not yet implemented — deferred as a future improvement.

### Each cgo package needs its own EKEventStore
Decided against shared `internal/store/` — C objects cannot be shared across cgo package boundaries. Each package (calendar, reminders) maintains its own `dispatch_once` singleton. Apple docs confirm multiple EKEventStore instances are fine.

## Decisions Made
- **All reminder writes via EventKit**: Unlike rem which uses AppleScript for writes, go-eventkit uses `saveReminder:commit:error:` and `removeReminder:commit:error:` directly. Eliminates subprocess overhead entirely.
- **No internal/store sharing**: Removed `internal/store/` package. Each package has its own singleton — simpler, avoids cross-cgo-package C object sharing issues.
- **Concurrency changes deferred**: The TLS error fix and write serialization queue are real improvements but not critical for the initial release. Both bridges work correctly in single-goroutine usage (the common case). Filed as future work.
- **Types always portable**: `reminders.go` and `calendar.go` have no build constraints, matching rem's `types.go` pattern. Types are importable on any platform.

## Key Insights

### No surprises — the patterns held
The existing JSON bridge pattern (ObjC → char* JSON → Go raw struct → Go typed struct) extended cleanly to recurrence rules and structured locations. The `rawRecurrenceRule` → `RecurrenceRule` conversion follows the exact same nullable-pointer pattern as other fields.

### EKRecurrenceRule complex initializer is the only way
EventKit's `EKRecurrenceRule` has two initializers: a simple one (frequency + interval + end) and a complex one (all constraint arrays). The complex initializer accepts nil for unused constraint arrays, so it works for all cases — even simple daily rules. Used this exclusively to avoid branching.

### `@(e.allDay)` was already wrong
While implementing `isDetached` serialization with proper `? @YES : @NO` ternary, discovered that `@(e.allDay)` and `@(e.hasRecurrenceRules)` were using the raw `@(bool)` pattern that produces 0/1 instead of true/false. Fixed both. This didn't cause visible bugs because Go's `json.Unmarshal` treats 0/1 as valid bools, but it was technically incorrect JSON.

### Coverage jumped significantly
Coverage went from 58.9% to 69.5% (+10.6%). The new code is all in parse.go (testable) rather than bridge_darwin.go (untestable via go test), so the new functionality contributes fully to coverage.

### Structured location coordinates survive round-trip perfectly
Integration tests confirmed that lat/long coordinates set via `CLLocation` in the ObjC bridge survive the full round-trip: Go → JSON → ObjC → EventKit → ObjC → JSON → Go with no precision loss (verified 37.3349 / -122.009).

### Update with empty recurrence rules correctly clears recurrence
The update path uses `removeRecurrenceRule:` on all existing rules before adding new ones. Passing an empty slice correctly makes the event non-recurring. The nil vs empty slice distinction in `UpdateEventInput` (`*[]RecurrenceRule`) works as designed: nil = no change, empty slice = remove all.

## Decisions Made
- **Single complex initializer for all rules**: Used `initRecurrenceWithFrequency:interval:daysOfTheWeek:daysOfTheMonth:monthsOfTheYear:weeksOfTheYear:daysOfTheYear:setPositions:end:` for all rules, even simple ones. Simpler code, one path to maintain.
- **Pointer fields for optional location coordinates in raw type**: Used `*float64` for latitude/longitude/radius in `rawStructuredLocation` to distinguish "not set" from "zero" (Null Island at 0,0 is a valid coordinate).
- **SpanFutureEvents for recurring event cleanup**: Changed integration test cleanup to use `SpanFutureEvents` instead of `SpanThisEvent` to ensure all occurrences of recurring test events are deleted.
- **No `firstDayOfTheWeek` in Go types**: Omitted this EKRecurrenceRule property as it's locale-dependent (0=system default) and rarely needed by consumers. Can be added later if requested.

### AppleScript variance increases dramatically with more iterations
Initial 10-iteration runs showed 286ms/201ms/2.87s for AppleScript. Bumping to 20 iterations (with warmup) revealed 620ms/432ms/9.2s — significantly slower. This is real-world behavior: osascript subprocess startup cost, Calendar/Reminders app activation, and IPC overhead compound. go-eventkit medians stayed stable across iteration counts (0.2ms/9ms/44ms → 0.2ms/18ms/47ms).

### "sub-200ms" was underselling the library
Every calendar CRUD operation is under 10ms median. Even `Events(365 days)` at 103ms is the only operation above 50ms median for calendar. Reminders are 15-54ms. Changed the claim to "3000x faster than AppleScript" which is both more impressive and backed by data.

### `b.Loop()` works in Go 1.24+
Used `for b.Loop()` instead of `for i := 0; i < b.N; i++` in benchmark functions. This is the modern Go 1.24 benchmark iteration pattern.

### P95 tightened significantly with more iterations
With 20 iterations, P95 showed high variance (e.g., calendar.Events(7 days) P95=56ms vs median 8.8ms). With 100 iterations, P95 dropped to 9.3ms — the outliers were cold-start artifacts. More iterations = more trustworthy P95 numbers for the README.

## Decisions Made
- **100/50/20 iteration counts**: Reads get 100 iterations (cheap, stable), writes get 50 (create real EventKit items), AppleScript gets 20 (each iteration spawns a subprocess + potentially activates an app — 20 is enough for stable medians without being obnoxious).
- **Lifecycle approach for write benchmarks**: Create → operate → cleanup, not create-and-delete in each iteration. Avoids measuring cleanup overhead in write timings. Pre-create pools for delete benchmarks.
- **`[bench]` prefix on all test items**: Makes cleanup reliable — all benchmark-created items have a recognizable prefix for manual cleanup if the script crashes.
- **README restructured for impact**: AppleScript comparison moved above the fold (first thing after the code snippet). Benchmark numbers are the hook — API details come after.

The LICENSE file must be included in a tagged version for pkg.go.dev to pick it up. Tagged `v0.1.0` without LICENSE, then added LICENSE and tagged `v0.1.1`.
