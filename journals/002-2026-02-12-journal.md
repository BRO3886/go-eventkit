## Key Insights
- **CGColor not NSColor**: Calendar/list color uses `CGColorRef` via `CGColorCreateGenericRGB()` — avoids AppKit dependency for the color path
- **Array-wrapping pattern**: Single JSON object responses reuse existing `parseListsJSON`/`parseCalendarsJSON` by wrapping in `"[" + jsonResp + "]"` — no need for separate single-object parsers
- **`contains` helper**: Case-insensitive error message matching for sentinel error detection (ErrNotFound, ErrImmutable) from ObjC error strings
- **Helper duplication**: `find_source_by_name`, `parse_hex_color` duplicated in both packages' ObjC files — can't share across cgo boundaries (same reason as EKEventStore singleton duplication)
- **Immutability check**: `cal.isImmutable` must be checked before update/delete — Apple returns generic error otherwise. We check explicitly and return `ErrImmutable`.
- **Source fallback**: If no source name specified, use `defaultCalendarForNewEvents.source` (calendar) or `defaultCalendarForNewReminders.source` (reminders)
- **Coverage gap analysis**: After initial tests, detailed coverage analysis revealed gaps in parseOptionalTime, parseListsJSON error path, marshalRecurrenceRules constraint arrays. Added targeted tests to fill them. Remaining uncovered lines are unreachable `json.Marshal` error returns.

## Decisions Made
- **`*string` pointer pattern for optional update fields**: nil means "don't change", empty string means "set to empty". Consistent with existing event/reminder update patterns.
- **Sentinel errors per package**: `calendar.ErrImmutable` and `reminders.ErrImmutable` are separate — follows existing `ErrNotFound`/`ErrUnsupported` pattern.
- **Integration test lifecycle**: create → verify in list → update (rename+recolor) → create child item → delete child → delete container → verify gone. Comprehensive lifecycle coverage.
